Comparing and Combining

Programs make decisions by comparing values and combining the results. Python groups its operators into families: arithmetic (covered in Hour 5), comparison for questions like "is this bigger?", logical for combining conditions, membership for "is this in that collection?", and identity for "are these literally the same object?". Every filter and validation rule you will ever write is built from these.

Two behaviors worth understanding early: logical operators short-circuit (the second operand is only evaluated when needed), and comparisons can be chained (100 <= price <= 5000) just like in mathematics.

Syntax Reference

Comparison operators return booleans. Note that equality is == (two signs) — a single = is assignment, and confusing them is a beginner rite of passage.

Comparison operators — ==, !=, <, >
price = 899.00

print(price == 899.00)    # Output: True — equal
print(price != 500.00)    # Output: True — not equal
print(price > 1000)       # Output: False
print(price <= 899.00)    # Output: True

Chained comparisons read like math: a <= x <= b checks that x lies in the range in one clean expression.

Chained comparison — range checks
price = 899.00

in_catalog_range = 100 <= price <= 5000
print(in_catalog_range)      # Output: True

# Equivalent to: 100 <= price and price <= 5000

and requires both sides to be true; or requires at least one; not flips a boolean. Combine them to express real business rules.

Logical operators — and, or, not
balance = 4500
account_type = "SAVINGS"
is_active = True

eligible = balance >= 1000 and account_type == "SAVINGS" and is_active
print(eligible)                  # Output: True — all three conditions hold

overdrawn = balance < 0 or not is_active
print(overdrawn)                 # Output: False — neither condition holds

Logical operators short-circuit: and stops at the first falsy value, or at the first truthy one. This lets a cheap guard protect an expensive (or dangerous) check.

Short-circuiting — guards that prevent errors
txn_count = 0
total = 4500

# Without the guard, 4500 / 0 would raise ZeroDivisionError
average = txn_count > 0 and total / txn_count
print(average)     # Output: False — right side never evaluated

txn_count = 3
average = txn_count > 0 and total / txn_count
print(average)     # Output: 1500.0

in tests membership: substring in string, element in list, key in dict, element in set. On sets and dict keys it is O(1) — effectively instant at any size.

Membership — in and not in
valid_categories = {"ELEC", "FOOD", "CLOTH"}
sku = "RET-ELEC-0042"

print("ELEC" in sku)                 # Output: True — substring check
print("ELEC" in valid_categories)    # Output: True — set membership, O(1)
print("TOYS" not in valid_categories) # Output: True
print("@" in "priya@retailbank.com")  # Output: True — quick email sanity check

is asks whether two names point to the same object in memory, while == asks whether their values are equal. Use == for data, is only for None.

Identity — is vs ==
x = [1, 2]
y = [1, 2]

print(x == y)     # Output: True — equal values
print(x is y)     # Output: False — two distinct list objects

phone = None
print(phone is None)     # Output: True — the correct None test

Edge Cases & Gotchas

and/or actually return one of their operands, not always a strict boolean. or is commonly used to supply defaults — but it treats every falsy value as "missing," which can bite with legitimate zeros.

Gotcha — or-defaults swallow zeros
requested_limit = 0        # customer explicitly asked for a 0 limit

limit = requested_limit or 5000    # 0 is falsy, so the default kicks in
print(limit)                        # Output: 5000 — the 0 was overridden!

# Safe version: only replace when the value is actually missing
limit = requested_limit if requested_limit is not None else 5000
print(limit)                        # Output: 0

Real-World Application

  • Filtering — chained comparisons express amount ranges; and/or route records to fraud vs normal queues.
  • Validation — in against an allowed set checks account types before approval; "@" in email is a first-pass sanity check.
  • Safety — short-circuit guards prevent division-by-zero and missing-key crashes in one line.

Key Takeaways

  • Comparisons return booleans and can be chained: a <= x <= b.
  • and/or short-circuit — use cheap guards before risky expressions.
  • in is O(1) on sets and dict keys — the right tool for lookups and allowlists.
  • == compares values; is compares identity — reserve is for None.
  • value or default replaces every falsy value, including a legitimate 0.
