Correlated Subqueries

Subqueries that reference outer rows — row-by-row comparisons for ranking, existence checks, and running analysis.

Go to Assignment →

A correlated subquery references a column from the outer query. Unlike a standalone subquery that runs once, a correlated subquery re-executes for each row of the outer query — comparing each row's value to its group average, or checking whether a related row exists.

Correlated vs. uncorrelated

Compare each row to a group average

SQL
-- hr.jobs priced above their category average
SELECT p.job_title, p.category, p.salary
FROM hr.jobs AS p
WHERE p.salary > (
  SELECT AVG(p2.salary)
  FROM hr.jobs AS p2
  WHERE p2.category = p.category
);

The inner query references p.category from the outer row — that is what makes it correlated.

EXISTS with correlation

SQL
-- hr.employees with at least one high-value sale
SELECT c.employee_id, c.last_name
FROM hr.employees AS c
WHERE EXISTS (
  SELECT 1
  FROM hr.employees AS s
  WHERE s.employee_id = c.employee_id
    AND s.salary > 500
);

-- hr.employees with no hr.job_history (NOT EXISTS)
SELECT a.employee_id, d.department_name
FROM hr.employees AS a
WHERE NOT EXISTS (
  SELECT 1
  FROM hr.job_history AS t
  WHERE t.employee_id = a.employee_id
);

Row-by-row ranking without window functions

SQL
-- Rows ranked within each partition
SELECT s1.employee_id, s1.department_id, s1.salary,
       (
         SELECT COUNT(*) + 1
         FROM hr.employees AS s2
         WHERE s2.department_id = s1.department_id
           AND s2.salary > s1.salary
       ) AS rank_in_partition
FROM hr.employees AS s1;

Window functions (Lessons 23–25) replace this pattern with cleaner, faster syntax — but correlated subqueries remain common in legacy codebases.

When to use JOIN instead

Many correlated subqueries can be rewritten as JOINs, which the optimizer often handles better:

SQL
-- Correlated EXISTS → often equivalent to INNER JOIN
SELECT DISTINCT c.employee_id, c.last_name
FROM hr.employees AS c
INNER JOIN hr.employees AS s
  ON s.employee_id = c.employee_id
 AND s.salary > 500;

Real-world context

  • capacity planning — Flag the jobs table priced above category average for competitive pricing review.
  • Credit review — Find the employees table whose latest loan application exceeds their average historical request.
  • jobs — Identify the departments table where today's sales exceed that store's 30-day daily average.

Practice

  1. Find the employees table whose total spend exceeds the overall group average (correlated scalar subquery).
  2. Use NOT EXISTS to list the jobs table never appearing in the job_history table.
  3. Rewrite one correlated EXISTS query as a JOIN and compare readability.

Key takeaways

  • Correlated subqueries reference outer query columns and re-run per outer row.
  • Ideal for row-vs-group comparisons and EXISTS/NOT EXISTS checks.
  • Can be slow on large datasets — consider JOINs or window functions as alternatives.
  • Understanding correlation is essential for reading legacy SQL in production pipelines.
