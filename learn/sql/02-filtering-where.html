Filtering with WHERE

Narrow millions of rows down to the records that matter — high-value job history records, employees in specific departments, or jobs in a salary range.

A large table might hold thousands of rows, and a detail table grows continuously. Scanning every row wastes compute and obscures the signal. The WHERE clause filters rows before they reach your result set — the same mechanism used in incremental jobs that process only a date partition.

Query execution order

Understanding this order explains why you cannot reference a SELECT alias inside WHERE — the filter runs before projection.

Compare values

Place WHERE after FROM. Text values use single quotes; numeric values do not.

SQL
-- Example: Sales under $50
SELECT job_title, salary
FROM hr.jobs
WHERE category = 'Sales'
  AND salary < 50;

-- Example: large transfers only
SELECT employee_id, employee_id, salary
FROM hr.job_history
WHERE job_id = 'transfer'
  AND salary >= 10000;

Combine conditions with AND / OR

AND requires every condition to be true. OR accepts a row when any condition is true. Parentheses prevent logic errors in complex filters.

SQL
-- employees in target departments in Mumbai or Delhi
SELECT employee_id, first_name, last_name, department_id
FROM hr.employees
WHERE department_id = '50'
  AND (department_id = 'Mumbai' OR department_id = 'Delhi');

-- Pending loan applications above threshold
SELECT application_id, employee_id, requested_amount
FROM hr.job_history
WHERE status = 'pending'
  AND requested_amount > 500000;

Match patterns and ranges

LIKE compares text patterns: % matches any length, _ matches one character. IN checks membership in a list. BETWEEN is inclusive on both ends.

SQL
-- jobs whose title starts with "EL-"
SELECT job_title, job_id
FROM hr.jobs
WHERE job_id LIKE 'EL-%';

-- Sales in Q1 2026
SELECT employee_id, hire_date, salary
FROM hr.employees
WHERE hire_date BETWEEN '2026-01-01' AND '2026-03-31';

-- employees with specific job titles
SELECT employee_id, job_title, salary
FROM hr.employees
WHERE job_title IN ('Sales Rep', 'Manager');

Handle missing values

NULL means unknown or not applicable — not zero, not empty string. Comparisons with = never match NULL.

SQL
-- employees missing a manager_id (data quality check)
SELECT employee_id, last_name
FROM hr.employees
WHERE Manager IS NULL;

-- Order items with a known discount
SELECT history_id, job_id, discount_pct
FROM hr.job_history
WHERE discount_pct IS NOT NULL;

Real-world context

  • ETL — Incremental loads use WHERE updated_at > :last_run to process only changed rows.
  • Anomaly detection — Filter the job_history table where salary exceeds a the group average by 3×.
  • capacity planning — Identify slow-moving job titles with WHERE hire_date < DATE('now', '-90 days') on joined tables.

Practice

  1. Find all jobs with min_salary between 10000 and 25000 inclusive.
  2. Return employees whose email ends in @example.com.
  3. List job_history rows with end_date IS NULL and salary above 65000.

Key takeaways

  • WHERE filters rows after FROM and before SELECT projection.
  • Parentheses make combined AND/OR logic reliable in production queries.
  • Use IS NULL / IS NOT NULL — never = NULL.
  • Filtering early reduces data scanned, which matters in enterprise scale.
