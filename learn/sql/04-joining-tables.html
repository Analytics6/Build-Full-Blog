Joining Tables

Combine related tables with INNER JOIN and LEFT JOIN — the core skill for building multi-table queries and reports.

Real datasets are normalized: one table holds master records, another holds detail rows, and lookup tables store reference data. A join connects tables through shared key columns, producing a single result set that applications and reports can consume.

INNER JOIN — matching rows only

An INNER JOIN returns rows only when a match exists on both sides. Rows without a partner are dropped.

SQL
-- Each employee with their department
SELECT e.employee_id, e.first_name, e.last_name, d.department_name
FROM hr.employees AS e
INNER JOIN hr.departments AS d
  ON e.department_id = d.department_id;

-- Job history enriched with job titles
SELECT oi.employee_id, p.job_title, oi.department_id, oi.salary
FROM hr.job_history AS oi
INNER JOIN hr.jobs AS p
  ON oi.job_id = p.job_id;

Table aliases

Aliases shorten references and are essential when joining three or more tables.

SQL
-- Three-table join: employee → job history → jobs
SELECT s.employee_id, p.job_title, oi.department_id
FROM hr.employees AS s
INNER JOIN hr.job_history AS oi ON oi.employee_id = s.employee_id
INNER JOIN hr.jobs AS p ON p.job_id = oi.job_id
WHERE s.hire_date >= '2026-01-01';

LEFT JOIN — keep every left row

A LEFT JOIN keeps all rows from the left table. Unmatched right-side columns become NULL.

SQL
-- All departments, including those with no employees
SELECT d.department_name, e.employee_id, e.salary
FROM hr.departments AS d
LEFT JOIN hr.employees AS e
  ON e.department_id = d.department_id;

-- Departments with no employees (anti-join pattern)
SELECT d.department_id, d.department_name
FROM hr.departments AS d
LEFT JOIN hr.employees AS e ON e.department_id = d.department_id
WHERE e.employee_id IS NULL;

Example: the employees table and the job_history table

SQL
-- Every employee with their job history count
SELECT e.employee_id, e.last_name, COUNT(jh.employee_id) AS history_count
FROM hr.employees AS e
LEFT JOIN hr.job_history AS jh ON jh.employee_id = e.employee_id
GROUP BY e.employee_id, e.last_name;

Real-world context

  • ETL — Fact tables join dimension tables before loading a warehouse dimensional model.
  • Complete records — LEFT JOIN preserves parent rows when optional child rows are missing.
  • Data quality — INNER JOIN between the job_history table and the jobs table exposes orphan foreign keys that fail the join.

Practice

  1. Join the job_history table to the jobs table and return job_title, department_id, and salary.
  2. List every department and its employees using LEFT JOIN; count how many have NULL employee_id.
  3. Join the employees table to the departments table and filter to department_id = 50.

Key takeaways

  • Joins relate tables through matching key columns (usually PK → FK).
  • INNER JOIN keeps matches only; LEFT JOIN preserves all left-side rows.
  • Aliases are essential for multi-table queries and self joins.
  • The anti-join pattern (LEFT JOIN … WHERE right.key IS NULL) finds rows without partners.
