Functions Basics

Subtopics: Defining functions, def keyword, parameter vs argument, return statement, return vs print

Write Once, Call Anywhere

The same calculation tends to appear again and again: a discount formula, a validation rule, a cleanup step. A function packages a block of code under a name so you write the logic once and call it wherever needed. Functions are the single most important tool for turning throwaway scripts into maintainable, testable programs.

Theory & Concepts

Anatomy of a Function

  • def name(parameters): starts the definition; the indented body follows.
  • A docstring — a triple-quoted string as the first line — documents what the function does.
  • return value hands a result back to the caller and ends the function immediately. A function without return returns None.

Parameters vs Arguments

Parameters are the placeholders in the definition (def calc_tax(amount, rate)); arguments are the actual values supplied at call time (calc_tax(1500.00, 0.18)). The distinction matters when reading error messages.

return vs print

print() only displays a value; the function still returns None. return sends the value back so it can be stored, compared, or passed onward. Rule of thumb: compute with return, display with print at the edges of the program.

Syntax Reference

A minimal function: define once with def, call as many times as you like.

def — defining and calling a function
def calc_order_total(quantity, unit_price):
    """Calculate the line total for one order line."""
    return quantity * unit_price

total = calc_order_total(3, 29.99)      # arguments: 3 and 29.99
print(total)                            # Output: 89.97

print(calc_order_total(10, 5.50))       # Output: 55.0 — reuse, no rewriting

The difference between returning and printing, made visible: only the returned value can be captured in a variable.

return vs print — capture vs display
def show_balance(balance):
    print(f"Balance: {balance}")        # displays, returns nothing

def get_balance_label(balance):
    return f"Balance: {balance}"        # returns, displays nothing

a = show_balance(5000)        # prints: Balance: 5000
b = get_balance_label(5000)   # prints nothing

print(a)     # Output: None — show_balance returned nothing usable
print(b)     # Output: Balance: 5000 — the value came back to us

return ends the function on the spot — any code after it never runs. This enables clean early-exit validation.

return — exits immediately
def classify_amount(amount):
    if amount > 50000:
        return "HIGH"          # function ends here for large amounts
    if amount > 10000:
        return "MEDIUM"
    return "LOW"               # reached only when both ifs failed

print(classify_amount(95000))   # Output: HIGH
print(classify_amount(500))     # Output: LOW

Returning several values packs them into a tuple; the caller unpacks them into separate names.

Multiple return values — tuple packing
def summarize_sales(prices):
    total = sum(prices)
    count = len(prices)
    return total, count, total / count      # three values, one tuple

total, count, avg = summarize_sales([29.99, 15.50, 42.00])

print(total)   # Output: 87.49
print(count)   # Output: 3
print(f"{avg:.2f}")   # Output: 29.16

Edge Cases & Gotchas

Variables created inside a function are local — they vanish when the function ends and are invisible outside. Trying to read one outside raises NameError.

Gotcha — local variables stay local
def apply_fee(balance):
    fee = 25.00                    # local to apply_fee
    return balance - fee

print(apply_fee(500))              # Output: 475.0
# print(fee)                      <-- NameError: name 'fee' is not defined

Real-World Application

  • Pipelines — transform_orders() standardizes columns once and is reused by every job that touches orders.
  • Composition — small functions like validate_iban() and flag_suspicious_amount() combine into larger fraud checks.
  • Testing — returned values can be asserted in unit tests; printed values cannot.

Key Takeaways

  • def names a reusable block; a docstring explains it; return hands back the result.
  • Parameters live in the definition; arguments are supplied at the call.
  • return beats print for anything downstream code needs — a printing function returns None.
  • return exits immediately, enabling early-exit validation chains.
  • Multiple returns are tuples; local variables never escape the function.
