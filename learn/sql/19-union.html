UNION and UNION ALL

Stack query results vertically — combine online and on-site relational data, or merge current and historical HR activity into one ledger view.

Combining Result Sets in HR

JOINs combine tables horizontally (side by side). UNION combines queries vertically (row on row). When the same logical columns come from different sources or time periods, you often need one unified result set — that is a UNION job.

Theory & Concepts

OperatorRemoves duplicates?When to use
UNIONYes (DISTINCT)When overlap exists and you want unique rows
UNION ALLNoWhen sets are disjoint or duplicates are acceptable — faster

Rules for UNION:

  • Each SELECT must return the same number of columns.
  • Corresponding columns must have compatible types.
  • Column names come from the first SELECT.
  • ORDER BY applies to the final combined result (place it after the last query).

Example: Combining Result Sets

SQL — Online + on-site revenue
SELECT
    'ONLINE' AS channel,
    history_id AS employee_id,
    employee_id,
    total AS salary,
    hire_date AS event_time
FROM hr.job_history
WHERE status = 'COMPLETED'

UNION ALL

SELECT
    'STORE' AS channel,
    employee_id AS employee_id,
    employee_id,
    department_id * salary AS salary,
    hire_date AS event_time
FROM hr.employees

ORDER BY event_time DESC;

Example: Stacked query results

SQL — Stacked result sets with UNION
SELECT
    employee_id,
    employee_id,
    salary,
    hire_date,
    'CHECKING' AS job_title
FROM hr.job_history

UNION ALL

SELECT
    employee_id,
    employee_id,
    salary,
    hire_date,
    'SAVINGS' AS job_title
FROM hr.job_history

ORDER BY hire_date;
SQL — UNION vs UNION ALL with duplicates
-- UNION removes the duplicate (1, 'Ada')
SELECT employee_id, name FROM hr.employees
UNION
SELECT employee_id, name FROM hr.employees;

-- UNION ALL keeps both rows — preferred when sets don't overlap
SELECT employee_id, name FROM hr.employees
UNION ALL
SELECT employee_id, name FROM hr.employees;

UNION vs JOIN

Real-World Notes

  • Prefer UNION ALL — When channels are mutually exclusive, skip the dedup sort that UNION performs; it's significantly faster on large fact tables.
  • Source tag column — Always add a literal column like 'STORE' / 'ONLINE' so downstream models know row origin.
  • Type alignment — Cast numeric and timestamp columns to matching types in both SELECT branches to avoid implicit cast failures.
  • Warehouse pattern — dbt models often UNION ALL staging tables from multiple source systems into a single fct_sales fact table.

Key Takeaways

  • UNION stacks compatible SELECT results vertically.
  • UNION ALL is faster when you don't need deduplication.
  • Column count and types must match across all UNION branches.
  • Add a source/channel column to preserve lineage in combined datasets.
