Why OOP Matters

Procedural scripts work for one-off tasks, but larger systems model real entities: customers, accounts, orders, transactions. Object-Oriented Programming (OOP) represents each entity as an object that bundles attributes (data) and methods (behavior). A BankAccount class encapsulates the account ID, the balance, and a withdraw() method — mirroring how production systems structure domain logic.

Theory & Concepts

Class vs Object

  • Class — blueprint/template (e.g., class StoreOrder)
  • Object (instance) — concrete entity created from the class (e.g., order #9001)

Attributes & Dot Notation

Access and set data on objects with dot notation: order.amount = 1500.00, print(customer.name).

Procedural vs OOP

Syntax Reference

The minimal class: class Name: plus a body (here just pass). Calling the class like a function creates an instance.

class — defining a blueprint and creating an object
class SavingsAccount:
    pass                        # empty blueprint for now

account = SavingsAccount()      # instantiation — a concrete object

print(type(account))            # Output: <class '__main__.SavingsAccount'>

Attributes attach data to an object via dot notation — set them with assignment, read them the same way.

Attributes — storing state with dot notation
class SavingsAccount:
    pass

account = SavingsAccount()
account.account_id = "ACC-8842"       # set attributes on the instance
account.balance = 15420.50

print(account.account_id)              # Output: ACC-8842
print(f"Balance: ${account.balance:,.2f}")
# Output: Balance: $15,420.50

Each instance holds its own independent data — two orders from the same class do not share attribute values.

Multiple instances — independent state
class StoreOrder:
    pass

order1 = StoreOrder()
order1.store_id = "S001"
order1.product = "Notebook"
order1.quantity = 5

order2 = StoreOrder()
order2.store_id = "S002"
order2.product = "Pen Set"
order2.quantity = 20

print(order1.store_id)    # Output: S001
print(order2.quantity)    # Output: 20 — order1's quantity is untouched

Real-World Application

  • ORM models — Django/SQLAlchemy map database tables to Python classes (Transaction, Product).
  • Data pipelines — Custom classes wrap validation rules for data records before load.
  • Analytics SDKs — Pandas Series/DataFrame are specialized objects with attributes and methods.

Key Takeaways

  • A class is a blueprint; an object is a specific instance created from it.
  • Attributes store object state; dot notation accesses them.
  • OOP models real-world entities — customers, orders, transactions — naturally.
  • Multiple instances of the same class hold independent data in memory.
