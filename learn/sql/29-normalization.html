Database Normalization

1NF, 2NF, 3NF and reducing redundancy — design clean schemas that avoid update anomalies and data drift.

Go to Assignment →

Why Normalize?

Imagine a single denormalized table storing an identifier, a person name, an email, a job title, and a salary on every row. If the email changes, you must update every row where it appears. If you miss one, reports show conflicting data. Normalization splits data into related tables so each fact is stored exactly once.

Theory: Normal Forms

FormRuleFixes
1NFAtomic values; no repeating groupsNo comma-separated value lists in one cell
2NF1NF + no partial dependencies on composite keysProduct name depends only on job_id, not order_id+job_id
3NF2NF + no transitive dependenciesStore city depends on department_id, not employee_id

Before: Denormalized Order Table

SQL — Problematic flat table
-- BAD: everything in one table
CREATE TABLE flat_employees (
    history_id       INTEGER,
    employee_name  TEXT,
    email TEXT,
    job_title   TEXT,
    salary  DECIMAL,
    department_id       INTEGER
);
-- Update anomaly: change email → update many rows
-- Insert anomaly: can't add child without parent
-- Delete anomaly: delete last child → lose parent ref

After: Normalized schema (3NF)

SQL — Normalized schema (3NF)
CREATE TABLE hr.employees (
    employee_id INTEGER PRIMARY KEY,
    name        TEXT NOT NULL,
    email       TEXT UNIQUE
);

CREATE TABLE hr.jobs (
    job_id   INTEGER PRIMARY KEY,
    job_id          TEXT UNIQUE,
    job_title TEXT,
    salary   DECIMAL
);

CREATE TABLE hr.job_history (
    history_id    INTEGER PRIMARY KEY,
    employee_id INTEGER REFERENCES hr.employees(employee_id),
    start_date  TIMESTAMP,
    status      TEXT
);

CREATE TABLE hr.job_history (
    history_id   INTEGER REFERENCES hr.job_history(history_id),
    job_id INTEGER REFERENCES hr.jobs(job_id),
    department_id   INTEGER,
    salary DECIMAL,
    PRIMARY KEY (history_id, job_id)
);

Example: Normalizing Branch Data

SQL — Separate branch attributes from hr.employees
-- 3NF: location stored once in departments, not repeated per employee
CREATE TABLE hr.departments (
    department_id   INTEGER PRIMARY KEY,
    branch_name TEXT,
    city        TEXT,
    region      TEXT
);

CREATE TABLE hr.employees (
    employee_id     INTEGER PRIMARY KEY,
    account_number TEXT UNIQUE,
    employee_id    INTEGER,
    department_id      INTEGER REFERENCES hr.departments(department_id),
    salary        DECIMAL
);

Normalization Decision Flow

Real-World Notes

  • OLTP vs OLAP — Normalize operational databases (OLTP source systems); denormalize data warehouses for query speed.
  • Surrogate keys — Use integer IDs as PKs even when natural keys (SKU, account number) exist — simplifies FK relationships.
  • Historical prices — Store salary on the job_history table (not just in catalog) to preserve price at time of sale.
  • Over-normalization — Splitting every attribute into its own table hurts JOIN performance; 3NF is the practical sweet spot for OLTP.

Key Takeaways

  • Normalization eliminates redundancy and update/insert/delete anomalies.
  • 1NF: atomic values; 2NF: no partial key dependencies; 3NF: no transitive dependencies.
  • Normalized designs use separate entity and junction tables.
  • Denormalize deliberately in analytics warehouses, not in source systems.
