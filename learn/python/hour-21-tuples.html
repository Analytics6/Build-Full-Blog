Immutable Records

A tuple is an ordered sequence that can never change after creation — a list with the mutability removed. That restriction is a feature: a tuple says "this is a fixed record," like a coordinate pair, a database row, or a (year, month, day) date. Because tuples are immutable and hashable, they can serve as dictionary keys, which lists cannot.

Theory & Concepts

Why Choose a Tuple over a List?

  • Safety — downstream code cannot accidentally modify a tuple; passing one to a function guarantees the caller's data survives intact.
  • Intent — a tuple signals a fixed-shape record (positions have meaning); a list signals a homogeneous, growable collection.
  • Hashability — tuples of immutable values can key dictionaries and live in sets.
  • Performance — tuples are slightly smaller and faster to create than lists.

Syntax Reference

Create tuples with parentheses — or nothing at all: it is actually the comma that makes a tuple. Indexing and slicing work exactly as with lists.

Creating tuples — ( ) literals
store_location = (40.7128, -74.0060)      # a lat/lon pair
txn_record = ("TXN-8842", "DEBIT", 4500.00)

print(store_location[0])     # Output: 40.7128 — indexing works like lists
print(txn_record[-1])        # Output: 4500.0
print(type(txn_record))      # Output: <class 'tuple'>

A single-element tuple needs a trailing comma — without it, the parentheses are just grouping and you get a plain value.

Single-element tuples — the trailing comma
single = (42,)         # a one-element tuple
not_a_tuple = (42)     # just the integer 42 in parentheses

print(type(single))        # Output: <class 'tuple'>
print(type(not_a_tuple))   # Output: <class 'int'>

Tuples are immutable — any attempt to assign into one raises TypeError. To "change" a tuple you build a new one.

Immutability — tuples cannot change
applied_date = (2026, 7, 16)       # year, month, day

# applied_date[0] = 2027    <-- TypeError: 'tuple' object does not support item assignment

updated = (2027,) + applied_date[1:]   # build a new tuple instead
print(updated)                         # Output: (2027, 7, 16)

Packing collects values into a tuple; unpacking spreads a tuple across variables. The variable count must match the element count.

Packing and unpacking — one-line assignment
txn = "TXN-8842", "DEBIT", 4500.00     # packing — the comma builds a tuple

txn_id, txn_type, amount = txn         # unpacking — three names at once

print(txn_id)     # Output: TXN-8842
print(txn_type)   # Output: DEBIT
print(amount)     # Output: 4500.0

Functions that "return multiple values" really return one tuple — the caller unpacks it. This idiom is everywhere in Python.

Multiple return values — tuples in disguise
def get_txn_summary():
    return 1500.00, "DEBIT", "TXN-8842"    # packs into one tuple

amount, txn_type, txn_id = get_txn_summary()   # unpack at the call site

print(amount)                 # Output: 1500.0
print(get_txn_summary())      # Output: (1500.0, 'DEBIT', 'TXN-8842')

count(value) returns how many times a value appears in the tuple — one of only two tuple methods.

tuple.count() — count occurrences
daily_statuses = ("OK", "OK", "FAIL", "OK")

print(daily_statuses.count("OK"))     # Output: 3
print(daily_statuses.count("FAIL"))   # Output: 1

index(value) returns the position of the first match, raising ValueError when the value is absent — the other tuple method.

tuple.index() — find a position
daily_statuses = ("OK", "OK", "FAIL", "OK")

print(daily_statuses.index("FAIL"))   # Output: 2 — first failure of the day
# daily_statuses.index("CRASH")       <-- ValueError: tuple.index(x): x not in tuple

tuple() converts any iterable into a tuple — the standard way to "freeze" a list so it cannot be changed afterwards.

tuple() — freeze a list
order_items = ["SKU-1", "SKU-2", "SKU-3"]

frozen_cart = tuple(order_items)     # locked at checkout for the audit trail

print(frozen_cart)           # Output: ('SKU-1', 'SKU-2', 'SKU-3')
# frozen_cart.append(...)    <-- AttributeError: no append on tuples

The generic sequence built-ins all work on tuples: len(), min(), max(), sum(), and the in operator.

len() / min() / max() — built-ins on tuples
weekly_sales = (1250, 980, 1450, 875, 1620, 1100, 990)

print(len(weekly_sales))    # Output: 7
print(min(weekly_sales))    # Output: 875
print(max(weekly_sales))    # Output: 1620
print(sum(weekly_sales))    # Output: 8265
print(1450 in weekly_sales) # Output: True

Edge Cases & Gotchas

Immutability is shallow: a tuple's slots cannot be reassigned, but a mutable object stored inside one can still be changed.

Gotcha — mutable objects inside tuples
record = ("ACC-8842", [1500.00, 2300.00])   # a list inside a tuple

record[1].append(999.00)      # legal! the tuple slot still holds the SAME list
print(record)                 # Output: ('ACC-8842', [1500.0, 2300.0, 999.0])

# record[1] = []              <-- TypeError: the slot itself cannot be reassigned

Real-World Application

  • Databases — SQL drivers return each row as a tuple: safe, fixed-shape records.
  • Function design — returning (value, status) pairs is cleaner than out-parameters.
  • Dictionary keys — (store_id, date) tuples key daily sales lookups.

Key Takeaways

  • Tuples are ordered and immutable; the comma (not the parentheses) creates them.
  • Single-element tuples require a trailing comma: (5,).
  • Packing/unpacking powers multi-assignment and multiple return values.
  • Only two methods exist — count() and index() — plus all the generic built-ins.
  • Immutability is shallow: a list inside a tuple can still be mutated.
