Self Joins Advanced

Join a table to itself — traverse department hierarchies and org charts, and manager-employee relationships within a single table.

When the Relationship Lives in One Table

Not every relationship needs a second table. In a sample schema, the employees table has a manager_id pointing to another row in the same table. the departments table may have parent_id. A self join aliases the table twice so each row can relate to another row in itself.

Theory & Concepts

A self join is a regular JOIN — you simply use two (or more) aliases for the same table:

SQL — Self join pattern
SELECT
    child.column,
    parent.column
FROM my_table AS child
JOIN my_table AS parent
  ON child.parent_id = parent.id;

Common patterns:

  • Hierarchy — parent/child categories, org charts
  • Sequential events — compare consecutive rows per group
  • Pairwise comparison — find the jobs table in the same category

Example: Category Hierarchy

SQL — Parent and child categories
SELECT
    child.category_id,
    child.category_name AS subcategory,
    parent.category_name AS parent_category
FROM hr.departments child
LEFT JOIN hr.departments parent
  ON child.parent_category_id = parent.category_id
ORDER BY parent.category_name, child.category_name;
SQL — hr.jobs with their parent category
SELECT
    p.job_id,
    p.job_title,
    cat.category_name,
    parent_cat.category_name AS department
FROM hr.jobs p
JOIN hr.departments cat ON p.category_id = cat.category_id
LEFT JOIN hr.departments parent_cat
  ON cat.parent_category_id = parent_cat.category_id;

Example: Manager Hierarchy

SQL — employee and their manager
SELECT
    e.employee_id,
    e.last_name AS employee,
    m.last_name AS manager,
    e.department_id
FROM hr.employees e
LEFT JOIN hr.employees m
  ON e.manager_id = m.employee_id
ORDER BY m.last_name, e.last_name;
SQL — Consecutive hr.job_history per account
SELECT
    curr.employee_id,
    curr.employee_id,
    curr.salary AS current_amount,
    prev.salary AS previous_amount,
    curr.hire_date
FROM hr.job_history curr
LEFT JOIN hr.job_history prev
  ON curr.employee_id = prev.employee_id
 AND prev.hire_date = (
    SELECT MAX(hire_date)
    FROM hr.job_history
    WHERE employee_id = curr.employee_id
      AND hire_date < curr.hire_date
 )
ORDER BY curr.employee_id, curr.hire_date;

Self Join Structure

Real-World Notes

  • Shallow hierarchies — Self joins work well for one level (employee → manager). Deeper trees need recursive CTEs (Topic 27).
  • Cartesian risk — Pairwise self joins without careful ON clauses explode row counts; always constrain the relationship.
  • Window functions alternative — For "previous row" comparisons, LAG() (Topic 25) is often cleaner than a self join.
  • Indexing — Index parent_id and manager_id foreign-key-like columns for fast hierarchy traversals.

Key Takeaways

  • A self join uses two aliases on the same table.
  • Ideal for hierarchies, manager chains, and row-to-row comparisons.
  • Use meaningful aliases (child/parent, curr/prev) for readability.
  • For multi-level trees, graduate to recursive CTEs.
