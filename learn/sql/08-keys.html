Primary Keys and Foreign Keys

Relate tables with key constraints — the relational backbone that makes joins between tables reliable.

Go to Assignment →

Keys give every row a stable identity and enforce relationships between tables. Without a primary key on a lookup table, you could not reliably join detail rows to descriptive labels. Without foreign keys, orphan rows would silently corrupt query results.

Key relationships in the sample schema

PK = Primary Key (unique identifier). FK = Foreign Key (references another table's PK).

Primary keys

A primary key uniquely identifies each row. It must be NOT NULL and unique. Common choices: auto-increment integer or UUID.

SQL
CREATE TABLE hr.employees (
  employee_id  INTEGER PRIMARY KEY,
  last_name    TEXT NOT NULL,
  email        TEXT UNIQUE
);

-- Composite PK on job_history
CREATE TABLE hr.job_history (
  employee_id      INTEGER NOT NULL,
  job_id   INTEGER NOT NULL,
  department_id     INTEGER NOT NULL,
  salary   DECIMAL(10, 2),
  PRIMARY KEY (employee_id, job_id)
);

Foreign keys

A foreign key column references a primary key in another table, enforcing referential integrity.

SQL
CREATE TABLE hr.employees (
  employee_id      INTEGER PRIMARY KEY,
  employee_id  INTEGER,
  salary DECIMAL(12, 2),
  FOREIGN KEY (employee_id) REFERENCES hr.employees(employee_id)
);

CREATE TABLE hr.job_history (
  employee_id INTEGER PRIMARY KEY,
  employee_id     INTEGER NOT NULL,
  salary         DECIMAL(15, 2),
  FOREIGN KEY (employee_id) REFERENCES hr.employees(employee_id)
);

Attempting to INSERT a employee_id that does not exist in the employees table will fail — protecting join integrity.

How keys enable joins

SQL
SELECT c.last_name, s.employee_id, s.salary
FROM hr.employees AS s
INNER JOIN hr.employees AS c
  ON s.employee_id = c.employee_id;

Example: employees → job_history → jobs

SQL
CREATE TABLE hr.job_history (
  application_id   INTEGER PRIMARY KEY,
  employee_id      INTEGER NOT NULL,
  requested_amount DECIMAL(12, 2),
  salary     INTEGER,
  FOREIGN KEY (employee_id) REFERENCES hr.employees(employee_id)
);

Real-world context

  • Data modeling — dimensional models use surrogate keys (auto IDs) as PKs on dimension tables.
  • ETL validation — FK violations during load signal source data problems before bad rows reach dashboards.
  • Surrogate vs. natural keys — Systems use integer IDs internally; job titles and account numbers are business keys with UNIQUE constraints.

Practice

  1. Add a FOREIGN KEY from job_history.job_id to jobs.job_id.
  2. Explain what happens if you INSERT an order_item with a nonexistent job_id when FK is enforced.
  3. Draw the PK/FK chain from employees → job_history → jobs.

Key takeaways

  • Primary keys uniquely identify rows; foreign keys link tables together.
  • FK constraints reject orphan rows at insert time.
  • Join conditions typically match FK columns to referenced PK columns.
  • Composite keys handle many-to-many junction tables like order line items.
