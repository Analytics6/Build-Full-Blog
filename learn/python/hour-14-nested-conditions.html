Decisions Inside Decisions

Real business rules rarely fit a single if. Loan underwriting checks the credit score, then the income, then the debt ratio — each check only meaningful after the previous one passed. Placing an if inside another if expresses exactly this gating. The craft lies in knowing when to nest and when to flatten the logic with and/or instead, because deep nesting quickly becomes unreadable.

Theory & Concepts

How Nesting Works

An inner if is just another statement inside the outer block, indented one more level (8 spaces total). It is evaluated only when the outer condition is true. Each level of nesting has its own optional else, so error messages can be specific about which gate failed.

Nest or Flatten?

  • Nest when the inner check depends on the outer one — checking a password only makes sense after the username exists.
  • Flatten with and when conditions are independent and only one combined outcome matters.
  • More than two or three levels is the "pyramid of doom" — refactor with combined conditions or early exits.

Syntax Reference

A three-gate underwriting rule: each inner check runs only after the outer one passes, and each gate has its own rejection message.

Nested if — loan underwriting gates
credit_score = 720
annual_income = 850000
loan_amount = 500000

if credit_score >= 650:                      # gate 1: creditworthiness
    if annual_income >= 300000:              # gate 2: income (only if gate 1 passed)
        dti = loan_amount / annual_income    # gate 3: debt-to-income
        if dti <= 0.6:
            decision = "APPROVED"
        else:
            decision = "REVIEW — high DTI"
    else:
        decision = "REJECTED — low income"
else:
    decision = "REJECTED — low credit score"

print(decision)   # Output: APPROVED

Nesting is the right shape for login-style checks: the inner question ("does the password match?") is meaningless unless the outer one ("does the user exist?") already passed.

Nested if — access control
username_exists = True
password_match = False

if username_exists:
    if password_match:
        print("Access granted — loading account statement")
    else:
        print("Invalid password")      # specific: we know the user exists
else:
    print("User not found")

# Output: Invalid password

When conditions are independent, flattening with and reads far better than nesting — compare this shipping rule to the pyramid it replaces.

Flattened logic — and instead of nesting
country = "IN"
weight_kg = 2.5
is_local = country == "IN"

if is_local and weight_kg <= 5:
    shipping_cost = 49
elif is_local and weight_kg > 5:
    shipping_cost = 99
else:
    shipping_cost = 499               # international orders

print(f"Shipping: {shipping_cost}")   # Output: Shipping: 49

A third option for deeply nested validation: early exits. Handle each failure immediately and let the happy path continue unindented — common in functions (Hour 26).

Early exits — flat validation chain
balance = 4500
withdrawal = 800
card_active = True

if not card_active:
    print("Card blocked")
elif withdrawal > balance:
    print("Insufficient funds")
else:
    balance -= withdrawal              # happy path — no nesting needed
    print(f"Dispensed {withdrawal}. New balance: {balance}")

# Output: Dispensed 800. New balance: 3700

Edge Cases & Gotchas

With nested blocks, an else pairs with the nearest unmatched if at the same indentation level. Misaligning it silently attaches it to the wrong condition.

Gotcha — which if does this else belong to?
amount = 500
is_verified = False

if amount > 100:
    if is_verified:
        print("Process large payment")
    else:                                  # pairs with the INNER if
        print("Verification required")

# Output: Verification required
# Outdent the else by 4 spaces and it would pair with the OUTER if,
# completely changing when it runs.

Real-World Application

  • Underwriting — layered credit, income, and collateral gates with gate-specific rejection reasons.
  • Access control — existence checks before credential checks, so error messages never leak information accidentally.
  • ETL routing — outer branch on source system, inner branch on record type.

Notebook & Practice

  • Write the three-gate loan example and test all four outcomes by varying the inputs.
  • Flatten a two-level nest into a single and condition and confirm identical behavior.
  • Rewrite a nested validation as an early-exit elif chain.
  • Shift an else between nesting levels and observe how the output changes.

Key Takeaways

  • An inner if runs only when its outer condition is true — perfect for dependent checks.
  • Flatten independent conditions with and/or; nest only when the inner check requires the outer gate.
  • Early-exit chains keep the happy path unindented and readable.
  • Beyond two levels of nesting, refactor — the pyramid of doom hides bugs.
  • An else pairs with the nearest if at its own indentation level.
