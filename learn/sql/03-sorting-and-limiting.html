Sorting and Limiting Results

Control result order with ORDER BY and return only the rows you need with LIMIT and OFFSET — essential for top-N reports and paginated dashboards.

Without an explicit sort, SQL returns rows in an undefined order — often insertion order, but never guaranteed. On a large employees table with millions of rows, "top 10 departments by total salary" requires ORDER BY and LIMIT. Pagination in a dashboard uses the same pattern with OFFSET.

Full SELECT execution order

Sort with ORDER BY

Ascending (ASC) is the default. Add DESC for descending order.

SQL
-- Lowest min_salary first
SELECT job_title, salary
FROM hr.jobs
ORDER BY salary ASC;

-- Largest hr.job_history first
SELECT employee_id, salary, hire_date
FROM hr.job_history
ORDER BY salary DESC;

Sort by multiple columns

SQL applies sort columns left to right — the first column is primary; later columns break ties.

SQL
-- Sales by store, then highest salary within each store
SELECT department_id, hire_date, salary
FROM hr.employees
ORDER BY department_id ASC, salary DESC;

-- Loan applications: status, then oldest first
SELECT application_id, status, submitted_at
FROM hr.job_history
ORDER BY status, submitted_at ASC;

Top-N with LIMIT

Always place LIMIT after ORDER BY so you limit the sorted slice, not arbitrary rows.

SQL
-- Top 5 highest-revenue sales
SELECT employee_id, department_id, salary
FROM hr.employees
ORDER BY salary DESC
LIMIT 5;

-- Most recent transaction per preview
SELECT employee_id, hire_date, salary
FROM hr.job_history
ORDER BY hire_date DESC
LIMIT 1;

Pagination with OFFSET

OFFSET skips rows before LIMIT returns its slice. Page 3 with 20 rows per page: LIMIT 20 OFFSET 40.

SQL
-- Page 3 of hr.employees (20 per page), newest first
SELECT employee_id, last_name, created_at
FROM hr.employees
ORDER BY created_at DESC, employee_id DESC
LIMIT 20 OFFSET 40;

Add a unique tie-breaker (like employee_id) to prevent rows from shifting between pages when sort keys collide.

Real-world context

  • Executive dashboards — "Top 10 job titles by units sold" uses ORDER BY + LIMIT on aggregated the job_history table.
  • API pagination — The assignment API returns 50 rows per request with OFFSET-based paging (cursor-based paging is preferred at very large scale).
  • Data sampling — Data scientists preview 1,000 random or recent rows before pulling a full training set.

Practice

  1. Return the ten most recent the employees table rows, newest first.
  2. List the jobs table items alphabetically by category, then by price descending.
  3. Write the query for page 4 when each page contains 15 the employees table rows.

Key takeaways

  • ORDER BY is the only way to guarantee result order.
  • Use multiple sort columns with a unique tie-breaker for stable pagination.
  • Apply LIMIT and OFFSET after sorting for reliable top-N and paging.
  • Sorting large tables is expensive — combine with WHERE filters to reduce work.
