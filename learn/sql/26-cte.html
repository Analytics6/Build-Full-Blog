CTEs: Named Subqueries at the Top

A Common Table Expression (CTE) is a temporary named result set defined with WITH and referenced in the main query. Instead of a 200-line nested subquery, you can write readable pipelines: first aggregate by group, then rank the results, then filter to the top rows.

Theory & Concepts

SQL — Basic CTE structure
WITH cte_name AS (
    SELECT ...
    FROM ...
)
SELECT *
FROM cte_name
WHERE ...;

Multiple CTEs chain with commas:

SQL — Chained CTEs
WITH
step_one AS (SELECT ...),
step_two AS (SELECT ... FROM step_one)
SELECT * FROM step_two;

CTEs improve readability and can be referenced multiple times in the same query. They are not automatically materialized — the optimizer may inline them.

Example: Multi-Step Aggregation

SQL — Store performance with CTE pipeline
WITH
dept_totals AS (
    SELECT
        department_id,
        SUM(department_id * salary) AS revenue
    FROM hr.employees
    WHERE hire_date >= '2026-01-01'
    GROUP BY department_id
),
dept_ranked AS (
    SELECT
        sr.department_id,
        d.department_name,
        sr.total_amount,
        RANK() OVER (ORDER BY sr.total_amount DESC) AS amount_rank
    FROM dept_totals sr
    JOIN hr.departments s ON s.department_id = sr.department_id
)
SELECT department_name, total_amount, amount_rank
FROM dept_ranked
WHERE amount_rank <= 10;

Example: High-Value Employee Summary

SQL — High-value employees with recent activity
WITH
employee_totals AS (
    SELECT
        employee_id,
        SUM(salary) AS total_volume,
        COUNT(*) AS history_count,
        MAX(hire_date) AS last_activity
    FROM hr.job_history
    GROUP BY employee_id
),
high_value AS (
    SELECT *
    FROM employee_totals
    WHERE total_volume > 100000
)
SELECT
    a.employee_id,
    a.last_name,
    hv.total_volume,
    hv.history_count,
    hv.last_activity
FROM high_value hv
JOIN hr.employees a ON a.employee_id = hv.employee_id
WHERE hv.last_activity >= CURRENT_DATE - INTERVAL '30 days'
ORDER BY hv.total_volume DESC;

Real-World Notes

  • dbt models — Each dbt model is essentially a persisted CTE; the ref() function chains them like WITH blocks.
  • Readability — Name CTEs after what they represent: monthly_totals, not temp1.
  • Reuse in one query — Unlike subqueries, a CTE can be referenced multiple times without duplicating logic.
  • Materialized CTEs — PostgreSQL supports MATERIALIZED hint when the optimizer re-scans a CTE expensively.

Key Takeaways

  • CTEs are named temporary result sets defined with WITH ... AS.
  • Chain multiple CTEs to build readable multi-step pipelines.
  • Prefer descriptive CTE names that document each transformation step.
  • Foundation for recursive CTEs (Topic 27) and modern analytics SQL.
