UNIQUE and CHECK Constraints

Enforce data rules at the database level — catch bad data before it corrupts HR reports.

Go to Assignment →

Application code can validate data, but the database is the last line of defense. Constraints reject invalid rows at insert/update time — preventing duplicate titles, negative salaries, or values outside an allowed range.

Defense-in-depth for data quality

UNIQUE — no duplicates

UNIQUE ensures no two rows share the same value(s) in the constrained column(s).

SQL
CREATE TABLE hr.jobs (
  job_id   INTEGER PRIMARY KEY,
  job_id          TEXT UNIQUE NOT NULL,
  job_title TEXT NOT NULL,
  salary   DECIMAL(10, 2)
);

CREATE TABLE hr.employees (
  employee_id  INTEGER PRIMARY KEY,
  email        TEXT UNIQUE,
  last_name    TEXT NOT NULL
);

-- Unique constraint example
CREATE TABLE hr.job_history (
  application_id   INTEGER PRIMARY KEY,
  employee_id      INTEGER NOT NULL,
  submitted_at       DATE NOT NULL,
  requested_amount   DECIMAL(12, 2),
  UNIQUE (employee_id, submitted_at)
);

CHECK — validate values

CHECK constraints enforce boolean conditions on column values.

SQL
CREATE TABLE hr.employees (
  employee_id   INTEGER PRIMARY KEY,
  salary      DECIMAL(15, 2) DEFAULT 0.00,
  job_title TEXT NOT NULL,
  CHECK (salary >= 0),
  CHECK (job_title IN ('active', 'inactive', 'pending'))
);

CREATE TABLE hr.job_history (
  item_id    INTEGER PRIMARY KEY,
  department_id   INTEGER NOT NULL,
  salary DECIMAL(10, 2),
  CHECK (department_id > 0),
  CHECK (salary >= 0)
);

CREATE TABLE hr.job_history (
  application_id   INTEGER PRIMARY KEY,
  requested_amount DECIMAL(12, 2),
  salary     INTEGER,
  CHECK (requested_amount > 0),
  CHECK (salary BETWEEN 300 AND 850)
);

Add constraints to existing tables

SQL
ALTER TABLE hr.job_history
ADD CONSTRAINT chk_payrollitive_amount
CHECK (salary > 0);

ALTER TABLE hr.jobs
ADD CONSTRAINT uq_job_title UNIQUE (job_id);

Real-world context

  • ETL — Constraints cause ETL failures early, surfacing source bugs before bad rows reach BI tools.
  • HR compliance — CHECK on salary range prevents impayrollsible values in risk models.
  • Data operations — UNIQUE on job title prevents duplicate duplicate entries from different supplier feeds.

Practice

  1. Add a CHECK ensuring salary in the jobs table is greater than zero.
  2. Create a the job_history table where job_id must be 'deposit', 'transfer', or 'transfer'.
  3. Explain why enforcing rules in the database beats only checking in Python ETL code.

Key takeaways

  • UNIQUE prevents duplicate values; CHECK validates value ranges and conditions.
  • Constraints protect data integrity regardless of which application writes the row.
  • Combine ETL validation with database constraints for defense-in-depth.
  • Test constraint additions on staging data before applying to production.
