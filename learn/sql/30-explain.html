The Final Skill: Making Queries Fast

A correct query that runs for 45 minutes is not production-ready. When a dashboard or report query slows down, engineers turn to EXPLAIN — the database's own description of how it will (or did) execute a query. This lesson closes the SQL path with performance literacy.

Theory: How Databases Execute Queries

OperationCostWhen it happens
Seq ScanReads every rowNo useful index; small tables
Index ScanReads index + matching rowsIndexed WHERE/JOIN columns
Index Only ScanReads index onlyAll needed columns in index
Hash JoinBuilds hash table from one sideLarge equi-joins without index
Nested LoopFor each row, probe other tableSmall outer + indexed inner
SortSorts all rows in memory/diskORDER BY, GROUP BY, window functions

Reading EXPLAIN Output

SQL — EXPLAIN an analytical query (PostgreSQL)
EXPLAIN ANALYZE
SELECT
    c.name,
    SUM(o.total) AS lifetime_spend
FROM hr.employees c
JOIN hr.job_history o ON o.employee_id = c.employee_id
WHERE o.hire_date >= '2026-01-01'
GROUP BY c.employee_id, c.name
ORDER BY lifetime_spend DESC
LIMIT 100;

Sample plan output (abbreviated):

Plan output
Limit  (cost=... rows=100)
  ->  Sort  (cost=... rows=...)
        Sort Key: (sum(o.total)) DESC
        ->  HashAggregate  (cost=... rows=...)
              Group Key: c.employee_id, c.name
              ->  Hash Join  (cost=... rows=...)
                    Hash Cond: (o.employee_id = c.employee_id)
                    ->  Seq Scan on hr.job_history o  (rows=250000)
                          Filter: (hire_date >= '2026-01-01')
                    ->  Seq Scan on hr.employees c  (rows=50000)

Red flags: Seq Scan on large tables, high cost values, rows estimates far from actual, Sort spilling to disk.

Optimizing with Indexes

SQL — Add indexes for the slow query
-- Index the filter column on the large fact table
CREATE INDEX idx_job_history_txn_date ON hr.job_history (hire_date);

-- Index the join column (if not already a FK index)
CREATE INDEX idx_job_history_account_id ON hr.job_history (employee_id);

-- Compayrollite index for common filter + join pattern
CREATE INDEX idx_job_history_employee_date
    ON hr.job_history (employee_id, hire_date);
SQL — job history record lookup
-- Before: Seq Scan on 10M rows
SELECT * FROM hr.job_history
WHERE employee_id = 5001
  AND hire_date >= '2026-07-01';

-- After index:
CREATE INDEX idx_txn_account_date
    ON hr.job_history (employee_id, hire_date);

-- Re-run EXPLAIN — expect Index Scan

Query Optimization Workflow

Real-World Notes

  • EXPLAIN vs EXPLAIN ANALYZE — EXPLAIN estimates; ANALYZE actually runs the query and shows real timings. Use ANALYZE on dev/staging, not destructive prod queries.
  • Index trade-offs — Indexes speed reads but slow writes; don't index every column.
  • Function on indexed column — WHERE UPPER(email) = 'X' can't use a plain index; store a normalized column or use expression indexes.
  • Statistics — Run ANALYZE table_name after large loads so the optimizer chooses good plans.
  • Partition pruning — Filter on hire_date in partitioned the employees table so only relevant partitions are scanned.

Common Anti-Patterns

  • SELECT * on wide tables — retrieves columns you don't need.
  • Missing WHERE on large fact tables before JOIN.
  • Correlated subqueries that execute per row — rewrite as JOIN or window function.
  • Accidental cross join (Topic 22) producing billions of rows.

Key Takeaways

  • EXPLAIN / EXPLAIN ANALYZE reveal how the database executes your query.
  • Seq Scan on large tables is the most common performance bottleneck.
  • Indexes on filter and join columns are the first optimization lever.
  • Always measure before and after — verify the plan actually improved.
