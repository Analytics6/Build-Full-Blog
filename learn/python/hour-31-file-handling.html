File Handling

Subtopics: opening files (open()), read/write/append modes ('r', 'w', 'a'), with statement (context managers), reading lines (read, readline, readlines)

Go to Assignment →

Why File Handling Matters in Data Analytics

Before data lands in a warehouse or Pandas DataFrame, it lives in files — nightly source systems dumps, audit logs, warehouse stock snapshots. Python's open() function and the with statement are how ETL scripts read raw feeds, write transformed output, and append pipeline log entries without leaking file handles.

Theory & Concepts

File Modes

ModeBehaviorData Analytics Use Case
"r"Read (default)Load incoming CSV/text feed
"w"Write (overwrites)Generate cleaned export file
"a"AppendAdd lines to ETL log without erasing history

The with Statement

A context manager that automatically closes the file when the block exits — even if an exception occurs. Always prefer with open(...) as f: over manual f.close().

Reading Strategies

  • f.read() — entire file as one string
  • f.readline() — one line at a time
  • f.readlines() — all lines as a list
  • for line in f: — memory-efficient iteration (best for large logs)

Syntax Reference

Mode "w" creates the file (or wipes it if it exists) and writes fresh content. Remember the explicit \n — write() adds no newline on its own.

open("w") — write a new file
with open("etl_log.txt", "w") as f:
    f.write("2026-07-15 02:00 — Started retail ingest\n")

# etl_log.txt now contains exactly one line.
# Running this again would ERASE the file and start over.

Mode "a" appends to the end without touching existing content — the right mode for logs that must keep their history.

open("a") — append without erasing
with open("etl_log.txt", "a") as f:
    f.write("2026-07-15 02:05 — Loaded 14,230 rows\n")

# etl_log.txt now has two lines — the first one survived.

Mode "r" reads. Iterating the file object line by line is the memory-safe pattern — only one line is in memory at a time, so gigabyte logs are no problem.

open("r") — read line by line
with open("etl_log.txt", "r") as f:
    for line in f:
        print(line.strip())          # strip() removes the trailing \n

# Output:
# 2026-07-15 02:00 — Started retail ingest
# 2026-07-15 02:05 — Loaded 14,230 rows

Filtering while reading: scan a log for failure markers and surface only the matching lines.

Line scan — find FAILED transactions
with open("transactions.log", "r") as f:
    for line in f:
        if "FAILED" in line:
            print(f"Alert: {line.strip()}")

# Output (example):
# Alert: TXN-8842 FAILED — insufficient funds

read() pulls the whole file into one string — fine for small files, and convenient for whole-text operations like word counts.

read() — whole file at once
with open("txn_log.txt", "r") as f:
    content = f.read()               # one big string

word_count = len(content.split())
print(f"Words in log: {word_count}")
# Output: Words in log: 42 (depends on the file)

readline() returns one line per call; readlines() returns all lines as a list. Both are occasionally handy, but the for line in f: loop covers most needs.

readline() / readlines() — the other readers
with open("etl_log.txt", "r") as f:
    first = f.readline()             # just the first line
    rest = f.readlines()             # remaining lines as a list

print(first.strip())    # Output: 2026-07-15 02:00 — Started retail ingest
print(len(rest))        # Output: 1 — one line left after readline()

Real-World Application

  • ETL logging — Append timestamped status lines to pipeline.log after each stage completes.
  • Data feeds — Read fixed-width or plain-text POS dumps line-by-line before CSV normalization.
  • Config files — Parse .env or .conf settings that control ETL job parameters.

  • Compare reading with readlines() vs iterating for line in f on a multi-line file.

Key Takeaways

  • Use with open(...) so files close automatically — critical in long-running jobs.
  • Choose the right mode: read, write (overwrite), or append.
  • For large files, iterate line-by-line instead of loading everything into memory.
  • File I/O is the first step in most structured data pipelines before CSV/Pandas processing.
