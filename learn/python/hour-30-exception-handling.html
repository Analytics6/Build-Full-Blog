Exception Handling

Subtopics: try block, except block, catching specific exceptions (ZeroDivisionError, ValueError, FileNotFoundError), else block, finally block

Go to Assignment →

Why Exception Handling Matters in Data Analytics

Nightly ETL jobs read thousands of files — source systems exports, transaction feeds, and warehouse stock updates. Not every file arrives perfectly formatted. A missing CSV, a division-by-zero in a metric calculation, or invalid user input should not crash the entire pipeline. Exception handling lets scripts fail gracefully, log errors, and continue processing remaining batches.

Theory & Concepts

try / except

Wrap risky code in try. If an exception occurs, Python jumps to the matching except block instead of terminating the program.

Catching Specific Exceptions

Always catch the narrowest exception type you expect:

  • ValueError — bad conversion (e.g., int("abc"))
  • ZeroDivisionError — dividing by zero in ratio calculations
  • FileNotFoundError — missing input file in an ETL stage
  • KeyError — missing dictionary key in parsed JSON

else and finally

  • else — runs only when the try block succeeds (no exception raised).
  • finally — always runs, ideal for closing files, DB connections, or writing audit trail entries.

Syntax Reference

The full four-part structure: risky code in try, error handling in except, success-only code in else, and guaranteed cleanup in finally.

try / except / else / finally — safe numeric conversion
try:
    amount = float("149.99")             # risky: could be non-numeric text
    fee = amount * 0.02
except ValueError:
    print("Invalid amount — expected a number")
else:
    print(f"Processing fee: {fee:.2f}")   # runs only on success
finally:
    print("Validation step complete")     # always runs

# Output:
# Processing fee: 3.00
# Validation step complete

When the risky code does fail, execution jumps straight to the matching except — the rest of the try block is skipped and the program keeps running.

except ValueError — catching the failure path
try:
    amount = float("abc")                 # raises ValueError immediately
    fee = amount * 0.02                   # never reached
except ValueError:
    print("Invalid amount — expected a number")

print("Script continues normally")

# Output:
# Invalid amount — expected a number
# Script continues normally

A missing input file is an expected event in ETL — catch FileNotFoundError and skip the stage instead of killing the whole job.

FileNotFoundError — a resilient ETL stage
try:
    with open("retail_orders_20260715.csv", "r") as f:
        header = f.readline()
except FileNotFoundError:
    print("Feed not arrived — skipping retail stage")
else:
    print(f"Loaded header: {header.strip()}")

# Output (when the file is missing):
# Feed not arrived — skipping retail stage

Multiple except blocks handle different failures differently — here a ratio calculator distinguishes bad input from a zero denominator.

Multiple except blocks — one handler per error
def safe_ratio(numerator_text, denominator_text):
    try:
        return float(numerator_text) / float(denominator_text)
    except ValueError:
        return "ERROR: not a number"
    except ZeroDivisionError:
        return "ERROR: denominator is zero"

print(safe_ratio("500", "40"))    # Output: 12.5
print(safe_ratio("500", "abc"))   # Output: ERROR: not a number
print(safe_ratio("500", "0"))     # Output: ERROR: denominator is zero

Wrapping risky access in a small helper that returns None on failure is a common defensive pattern.

IndexError — safe list access helper
def safe_get(records, index):
    try:
        return records[index]
    except IndexError:
        return None

customers = ["Alice", "Bob"]

print(safe_get(customers, 0))    # Output: Alice
print(safe_get(customers, 5))    # Output: None — no crash

Real-World Application

  • ETL resilience — The ingest script catches FileNotFoundError per store and alerts ops instead of halting all 200 stores.
  • Data validation — ValueError on malformed IBAN strings routes rows to a quarantine table.
  • API services — finally closes database connections even when queries fail mid-transaction.

  • Chain multiple except blocks for ValueError and ZeroDivisionError in a ratio calculator.

Key Takeaways

  • Exceptions are runtime errors; handling them keeps production pipelines alive.
  • Catch specific exception types — avoid bare except: that hides bugs.
  • else runs on success; finally always runs for cleanup.
  • Robust ETL scripts expect bad data and recover instead of crashing.
