Aggregations

Summarize data with COUNT, SUM, and AVG, then group results with GROUP BY and filter groups using HAVING.

Raw rows in the employees table or the job_history table are too granular for summary reports. Aggregate functions collapse many rows into KPIs — totals, averages, and row counts — that power dashboards.

Aggregation pipeline

Common aggregate functions

FunctionReturnsExample
COUNT()number of rows/valuessales per store
SUM()total numeric valuedaily totals
AVG()mean numeric valueavg loan salary
MIN()/MAX()extremesearliest/latest txn
SQL
-- Example: total revenue across all sales
SELECT SUM(salary) AS total_amount,
       AVG(salary) AS avg_basket,
       COUNT(*) AS sale_count
FROM hr.employees;

-- Example: transaction volume
SELECT COUNT(*) AS history_count,
       SUM(salary) AS total_volume
FROM hr.job_history
WHERE job_id = 'deposit';

COUNT(*) counts every row; COUNT(column) skips NULLs in that column.

GROUP BY — one summary per bucket

Every non-aggregated column in SELECT must appear in GROUP BY.

SQL
-- Totals by group (via join)
SELECT p.category,
       SUM(oi.salary) AS category_total
FROM hr.job_history AS oi
INNER JOIN hr.jobs AS p ON p.job_id = oi.job_id
GROUP BY p.category
ORDER BY category_total DESC;

-- Sales count per store
SELECT department_id, COUNT(*) AS sales_count
FROM hr.employees
GROUP BY department_id;

HAVING — filter groups, not rows

WHERE filters individual rows before grouping. HAVING filters the groups afterward and can reference aggregates.

SQL
-- hr.departments with more than 100 sales
SELECT department_id, COUNT(*) AS sales_count
FROM hr.employees
GROUP BY department_id
HAVING COUNT(*) > 100;

-- hr.employees whose 2026 spend exceeds 500
SELECT employee_id, SUM(salary) AS spend_2026
FROM hr.employees
WHERE hire_date >= '2026-01-01'
GROUP BY employee_id
HAVING SUM(salary) > 500;

Aggregations with joins

SQL
-- Every parent row and child count (including zero)
SELECT c.last_name, COUNT(s.employee_id) AS employee_count
FROM hr.employees AS c
LEFT JOIN hr.employees AS s ON s.employee_id = c.employee_id
GROUP BY c.employee_id, c.last_name
ORDER BY employee_count DESC;

-- Average loan salary by status
SELECT status, AVG(requested_amount) AS avg_amount
FROM hr.job_history
GROUP BY status;

Real-world context

  • ETL validation — After loading the employees table, GROUP BY hire_date and compare row counts to source files.
  • Feature engineering — Aggregate per-entity totals from a detail table before training a churn model.
  • Risk reporting — HAVING on average transaction size flags the employees table with unusual activity patterns.

Practice

  1. Find the min, max, and average salary in the jobs table.
  2. Count the job_history table grouped by job_id.
  3. Find departments with total aggregated value over 10,000 using JOIN, GROUP BY, and HAVING.

Key takeaways

  • Aggregate functions summarize values across rows.
  • GROUP BY creates one summary row per distinct bucket.
  • WHERE filters rows; HAVING filters groups after aggregation.
  • Combine joins with aggregates for human-readable business reports.
