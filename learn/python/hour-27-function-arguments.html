Function Arguments

Subtopics: Positional arguments, keyword arguments, default arguments, variable-length arguments (*args, **kwargs)

Go to Assignment →

One Function, Many Call Styles

Real inputs vary: some invoices carry a discount, most do not; one report aggregates 3 stores, another 30. Python's argument system — positional, keyword, default, and variable-length — lets a single function absorb that variety gracefully. These are also the patterns behind every library signature you will read, from sorted(key=..., reverse=...) to Pandas.

Theory & Concepts

The Four Kinds

  • Positional — matched by order: calc(500, 0.18).
  • Keyword — matched by name, order-independent: calc(rate=0.18, amount=500).
  • Default — parameters with fallback values in the definition; callers may omit them.
  • Variable-length — *args collects extra positional values into a tuple; **kwargs collects extra named values into a dict.

Definition order is fixed: required parameters → defaults → *args → **kwargs. Defaults may never precede required parameters.

Syntax Reference

Positional arguments bind strictly by position — swap them and the function silently computes the wrong thing.

Positional arguments — order matters
def apply_discount(amount, rate):
    return round(amount * (1 - rate), 2)

print(apply_discount(500, 0.10))    # Output: 450.0 — correct
print(apply_discount(0.10, 500))    # Output: -49.99 — swapped, silently wrong!

Keyword arguments bind by name, so the order no longer matters and the call documents itself.

Keyword arguments — bind by name
def apply_discount(amount, rate):
    return round(amount * (1 - rate), 2)

print(apply_discount(amount=500, rate=0.10))    # Output: 450.0
print(apply_discount(rate=0.10, amount=500))    # Output: 450.0 — order-proof

Default parameters give a fallback value. Callers override only what differs from the common case.

Default parameters — optional with fallbacks
def calc_invoice(amount, tax_rate=0.18, discount=0.0):
    subtotal = amount * (1 - discount)
    return round(subtotal * (1 + tax_rate), 2)

print(calc_invoice(500))                    # Output: 590.0 — both defaults used
print(calc_invoice(500, discount=0.05))     # Output: 560.5 — override one by name
print(calc_invoice(500, 0.08, 0.10))        # Output: 486.0 — all positional

*args gathers any number of extra positional arguments into a tuple — ideal when the caller decides how many values to pass.

*args — variable positional arguments
def total_daily_sales(*amounts):
    print(type(amounts))            # inside, amounts is a tuple
    return sum(amounts)

result = total_daily_sales(1200, 980, 1450)
# Output: <class 'tuple'>

print(result)                       # Output: 3630 — three stores
print(total_daily_sales(500))       # Output: <class 'tuple'> then 500 — one store works too

**kwargs gathers extra keyword arguments into a dict — the pattern behind flexible metadata and option forwarding.

**kwargs — variable keyword arguments
def log_customer_event(event_type, **metadata):
    print(f"Event: {event_type}")
    for key, value in metadata.items():
        print(f"  {key}: {value}")

log_customer_event("purchase", customer_id=1042, category="Electronics", amount=92.50)

# Output:
# Event: purchase
#   customer_id: 1042
#   category: Electronics
#   amount: 92.5

A full signature combines all four kinds — in the mandatory order: required, defaults, *args, **kwargs.

Full signature — all four kinds together
def pipeline_step(source, target="warehouse", *extra_paths, **options):
    print(source)          # required positional
    print(target)          # default
    print(extra_paths)     # tuple of extras
    print(options)         # dict of named options

pipeline_step("pos.csv", "staging", "backup.csv", encoding="utf-8", skip_rows=1)

# Output:
# pos.csv
# staging
# ('backup.csv',)
# {'encoding': 'utf-8', 'skip_rows': 1}

Edge Cases & Gotchas

The most infamous Python trap: a mutable default (like []) is created once at definition time and shared across every call. Use None as the default and create the list inside.

Gotcha — mutable default arguments
def add_txn_bad(txn, batch=[]):        # ONE shared list for all calls!
    batch.append(txn)
    return batch

print(add_txn_bad("TXN-1"))    # Output: ['TXN-1']
print(add_txn_bad("TXN-2"))    # Output: ['TXN-1', 'TXN-2'] — leaked!

def add_txn_good(txn, batch=None):
    if batch is None:
        batch = []                     # fresh list per call
    batch.append(txn)
    return batch

print(add_txn_good("TXN-1"))   # Output: ['TXN-1']
print(add_txn_good("TXN-2"))   # Output: ['TXN-2'] — independent

Real-World Application

  • Billing — calc_invoice(amount, tax_rate=0.18) serves multiple tax jurisdictions with one function.
  • Wrappers — load_table(name, **db_options) forwards timeouts and batch sizes to the database driver untouched.
  • Frameworks — web frameworks and Airflow operators lean on *args, **kwargs so middleware can inject context.

Key Takeaways

  • Positional arguments bind by order; keyword arguments bind by name and self-document.
  • Defaults make parameters optional; they must follow required parameters.
  • *args → tuple of extra positionals; **kwargs → dict of extra keywords.
  • Signature order: required → defaults → *args → **kwargs.
  • Never use a mutable value as a default — use None and create it inside.
