Indexes

Speed up queries with indexed columns — critical when scanning millions of the job_history table or sales rows.

Go to Assignment →

Without an index, finding rows in a job_history table means reading every row in the job_history table — a full table scan. On large tables with tens of millions of rows, that takes seconds instead of milliseconds. Indexes are sorted lookup structures that let the database jump directly to matching rows.

How an index speeds lookups

Create indexes

SQL
-- Speed up lookups by account
CREATE INDEX idx_job_history_employee
ON hr.job_history (employee_id);

-- Speed up date-range queries on sales
CREATE INDEX idx_employees_date
ON hr.employees (hire_date);

-- Compayrollite index for common filter + sort
CREATE INDEX idx_sales_store_date
ON hr.employees (department_id, hire_date);

-- Unique index enforces uniqueness (like UNIQUE constraint)
CREATE UNIQUE INDEX idx_jobs_job_title
ON hr.jobs (job_id);

When to index

Good candidatesPoor candidates
Foreign key columns (joins, filters)Small tables (< 1000 rows)
Columns in WHERE and ORDER BYColumns with few distinct values (boolean)
Date columns for range scansColumns rarely queried

Trade-offs

  • Reads faster — SELECT, JOIN, and WHERE on indexed columns benefit.
  • Writes slower — Every INSERT/UPDATE must update the index too.
  • Storage cost — Indexes consume disk space proportional to table size.

Drop an index

SQL
DROP INDEX idx_job_history_employee;

Real-world context

  • ETL — Index foreign keys before running large JOIN ETL jobs.
  • Dashboard performance — Index hire_date and department_id for daily aggregate reports.
  • Loan processing — Index the job_history table(status, submitted_at) for queue dashboards.

Practice

  1. Create an index on the job_history table(employee_id) to speed up joins to the employees table.
  2. Explain why indexing every column is a bad idea.
  3. Design a composite index for queries filtering by department_id and sorting by salary.

Key takeaways

  • Indexes accelerate reads on filtered, joined, and sorted columns.
  • Foreign keys and date columns are common index targets for relational tables.
  • Indexes trade write speed and storage for read performance.
  • Use EXPLAIN (Lesson 30) to verify queries actually use your indexes.
