Window Functions — LAG and LEAD

Access previous and next row values — compute month-over-month payroll spend growth and day-over-day HR salary changes without self joins.

Go to Assignment →

Time-Travel Within a Result Set

LAG() looks backward; LEAD() looks forward. For each row, they fetch a value from a prior or subsequent row in the same partition. This replaces awkward self joins when computing period-over-period changes in sales and transaction data.

Theory & Concepts

SQL — LAG and LEAD syntax
LAG(column, offset, default) OVER (
    PARTITION BY group_col
    ORDER BY sort_col
)

LEAD(column, offset, default) OVER (
    PARTITION BY group_col
    ORDER BY sort_col
)
ParameterDefaultMeaning
offset1How many rows back (LAG) or forward (LEAD)
defaultNULLValue when no prior/next row exists

Example: Period-over-Period Comparison

SQL — Revenue growth by month
WITH monthly_totals AS (
    SELECT
        DATE_TRUNC('month', hire_date) AS report_month,
        SUM(department_id * salary) AS revenue
    FROM hr.employees
    GROUP BY 1
)
SELECT
    report_month,
    revenue,
    LAG(revenue) OVER (ORDER BY report_month) AS prev_month_total,
    revenue - LAG(revenue) OVER (ORDER BY report_month) AS monthly_change,
    ROUND(
        100.0 * (revenue - LAG(revenue) OVER (ORDER BY report_month))
        / NULLIF(LAG(revenue) OVER (ORDER BY report_month), 0),
        2
    ) AS pct_change
FROM monthly_totals
ORDER BY report_month;

Example: Transaction Sequence Analysis

SQL — Balance change between consecutive hr.job_history
SELECT
    employee_id,
    employee_id,
    hire_date,
    salary,
    LAG(salary) OVER (
        PARTITION BY employee_id
        ORDER BY hire_date
    ) AS prev_amount,
    salary - LAG(salary) OVER (
        PARTITION BY employee_id
        ORDER BY hire_date
    ) AS amount_delta,
    LEAD(hire_date) OVER (
        PARTITION BY employee_id
        ORDER BY hire_date
    ) AS next_txn_time
FROM hr.job_history
ORDER BY employee_id, hire_date;
SQL — Days between consecutive rows
SELECT
    employee_id,
    history_id,
    hire_date,
    LAG(hire_date) OVER (
        PARTITION BY employee_id
        ORDER BY hire_date
    ) AS prev_order_date,
    hire_date - LAG(hire_date) OVER (
        PARTITION BY employee_id
        ORDER BY hire_date
    ) AS days_since_last_order
FROM hr.job_history
WHERE status = 'COMPLETED';

LAG / LEAD Direction

Real-World Notes

  • Period comparisons — LAG is the 30 for MoM, WoW, and YoY metrics in BI dashboards without exporting to Python.
  • Session gaps — Use LAG on timestamps to detect inactivity gaps between payroll sessions or employee login events.
  • Default values — Pass a third argument to LAG/LEAD (e.g., 0) to avoid NULL arithmetic on the first row.
  • Offset > 1 — LAG(revenue, 12) compares to the same month last year in monthly data.

Key Takeaways

  • LAG() retrieves a value from a preceding row in the partition.
  • LEAD() retrieves a value from a following row.
  • ORDER BY inside OVER is mandatory and defines "preceding" vs "following."
  • Ideal for period-over-period change, gaps, and sequential analysis.
