Python Syntax

Subtopics: Print statement, comments, indentation, basic blocks

Go to Assignment →

Why Syntax Matters

Syntax is the grammar of a programming language. Python's most distinctive rule is that indentation is not decoration — it is structure. Where other languages use braces { } to group statements, Python uses whitespace. A single misplaced space can change what a program does or stop it from running at all, so mastering these rules early prevents frustrating errors later.

Theory & Concepts

Indentation Defines Blocks

After any line ending in a colon (:) — an if, a loop, a function definition — the following lines must be indented to form the block that belongs to it. The convention is 4 spaces per level. When the indentation returns to the previous level, the block ends.

Common pitfalls:

  • Mixing tabs and spaces raises TabError or IndentationError. Configure your editor to insert 4 spaces when you press Tab.
  • Forgetting to indent after a colon raises IndentationError: expected an indented block.
  • Indenting a line that should not be indented silently changes the logic — the statement becomes part of the block above it. This is a bug, not an error, so Python will not warn you.

Comments

  • Single-line — start with #; everything after it on that line is ignored by the interpreter.
  • Multi-line — Python has no true block-comment syntax; the convention is a triple-quoted string """ ... """, which also serves as a docstring when placed at the top of a function or module.

Best practice: comment the why, not the what. # add 1 to counter restates the code; # retry once because the API is flaky explains a decision.

The print() Function

print() writes values to the console and accepts several values at once, plus two useful keyword parameters:

ParameterDefaultPurpose
sep' ' (space)Separator placed between multiple values
end'\n' (newline)What to print after the last value

Syntax Reference

Example — Indentation creates a block
# Log pipeline stages only when the pipeline is enabled
pipeline_enabled = True

if pipeline_enabled:
    print("Extracting retail inventory CSV...")   # inside the if-block (4 spaces)
    print("Rows loaded: 15420")                    # still inside the block

print("Pipeline check finished.")                   # outside — always runs

# Output:
# Extracting retail inventory CSV...
# Rows loaded: 15420
# Pipeline check finished.
Example — A missing indent is an error
# WRONG — the line after the colon must be indented
# if pipeline_enabled:
# print("Missing indent")     <-- IndentationError: expected an indented block

# WRONG in a different way — extra indent changes MEANING, not validity
account_balance = 900
if account_balance > 1000:
    print("High balance")
    print("Sending premium offer")   # runs ONLY when balance > 1000

# If that last print were unindented, the offer would be sent to everyone —
# a logic bug the interpreter cannot catch for you.

The sep parameter controls what appears between the values you pass to print() — the default is a single space.

Example — print() with sep
# Format one transaction record on a single log line
account_id = "ACC-8842"
amount = 1500.00
txn_type = "DEBIT"

print(account_id, amount, txn_type, sep=" | ")
# Output: ACC-8842 | 1500.0 | DEBIT

print("retail", "sales", "report", sep="*")
# Output: retail*sales*report

The end parameter replaces the newline that normally follows each print() — useful for building a line across several calls.

Example — print() with end
print("Processing", end=" ")      # end=" " keeps the cursor on the same line
print("complete.")

# Output: Processing complete.

Single-line comments start with # — everything after it on the line is ignored, whether the comment fills the line or trails a statement.

Example — single-line comments
# Single-line comment: explain intent, not mechanics
daily_revenue = 48250.75   # trailing comment: sum of store_sales for today

print(daily_revenue)       # Output: 48250.75

Triple-quoted strings serve as multi-line comments and docstrings — Python evaluates them as string literals but ignores them when they stand alone.

Example — multi-line docstring comment
"""
This triple-quoted string acts as a multi-line comment.
It documents the aggregation step below for the analytics team:
we divide completed orders by total orders to get a fill rate.
"""
orders_completed = 940
orders_total = 1000
fill_rate = orders_completed / orders_total

print(f"Fill rate: {fill_rate:.0%}")   # Output: Fill rate: 94%

Real-World Application

  • Pipeline logging — print() with end=" ... " streams progress messages while a long ingestion job runs.
  • Readable code reviews — teams enforce 4-space indentation with tools like black and flake8 so every script looks consistent.
  • Docstrings — triple-quoted strings at the top of functions power auto-generated documentation and editor tooltips.

Notebook & Practice

  • Write an if True: block with two indented print() statements, then add a third unindented print and observe when each runs.
  • Print three words separated by asterisks using sep='*'.
  • Print two sentences on the same line using end=' '.
  • Deliberately remove an indent after a colon and read the IndentationError message.

Key Takeaways

  • Python uses indentation (4 spaces per level) instead of braces to define code blocks — it is part of the syntax, not style.
  • A wrong indent can be a hard error (IndentationError) or, worse, a silent logic bug.
  • # starts single-line comments; triple-quoted strings serve as multi-line comments and docstrings.
  • print() accepts sep and end to control output formatting without string concatenation.
