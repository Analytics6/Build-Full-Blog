Recursive CTEs

Query hierarchical data like org charts and category trees — traverse multi-level department hierarchies and management chains in SQL.

Go to Assignment →

Beyond One-Level Self Joins

A manager's manager's manager requires traversing multiple levels. Recursive CTEs solve this in SQL by combining an anchor query (the starting point) with a recursive query that keeps joining until no new rows appear. HR chains use them for category trees, bill-of-materials, and organizational hierarchies.

Theory & Concepts

SQL — Recursive CTE template
WITH RECURSIVE cte_name AS (
    -- Anchor: starting rows
    SELECT id, parent_id, name, 1 AS depth
    FROM hierarchy
    WHERE parent_id IS NULL

    UNION ALL

    -- Recursive: join to previous iteration
    SELECT h.id, h.parent_id, h.name, c.depth + 1
    FROM hierarchy h
    JOIN cte_name c ON h.parent_id = c.id
)
SELECT * FROM cte_name;

Key rules:

  • Must use WITH RECURSIVE (in PostgreSQL and SQLite).
  • Anchor and recursive parts connected by UNION ALL.
  • Recursive part references the CTE name — that's the recursion.
  • Always include a transfer condition to avoid infinite loops.

Example: Organizational Hierarchy

SQL — All category levels with depth
WITH RECURSIVE category_tree AS (
    SELECT
        category_id,
        category_name,
        parent_category_id,
        category_name AS path,
        1 AS depth
    FROM hr.departments
    WHERE parent_category_id IS NULL

    UNION ALL

    SELECT
        c.category_id,
        c.category_name,
        c.parent_category_id,
        ct.path || ' > ' || c.category_name,
        ct.depth + 1
    FROM hr.departments c
    JOIN category_tree ct ON c.parent_category_id = ct.category_id
)
SELECT category_id, path, depth
FROM category_tree
ORDER BY path;

Example: Manager Chain

SQL — All reports under a branch manager
WITH RECURSIVE org_chart AS (
    -- Start with the branch manager
    SELECT
        employee_id,
        last_name,
        manager_id,
        1 AS level
    FROM hr.employees
    WHERE employee_id = 42  -- branch manager

    UNION ALL

    SELECT
        e.employee_id,
        e.last_name,
        e.manager_id,
        oc.level + 1
    FROM hr.employees e
    JOIN org_chart oc ON e.manager_id = oc.employee_id
)
SELECT employee_id, last_name, level
FROM org_chart
ORDER BY level, last_name;
SQL — Generate a number sequence (non-hierarchy use)
WITH RECURSIVE nums AS (
    SELECT 1 AS n
    UNION ALL
    SELECT n + 1 FROM nums WHERE n < 31
)
SELECT n AS day_of_month FROM nums;

Real-World Notes

  • Depth limits — Add WHERE depth < 10 as a safety guard against cyclic parent_id data.
  • Cycle detection — PostgreSQL offers CYCLE clause; otherwise validate hierarchies in ETL before querying.
  • Date spines — Recursive CTEs generate number/date sequences when no calendar table exists.
  • Graph databases — Very deep or highly connected graphs may be better suited to Neo4j; recursive CTEs handle typical org and category trees well.

Key Takeaways

  • Recursive CTEs combine an anchor query with a self-referencing recursive step.
  • Ideal for hierarchies: org charts, categories, bill-of-materials.
  • Track depth or level to limit traversal and display indentation.
  • Also useful for generating sequences without a numbers table.
