Views

Save queries as reusable virtual tables — simplify analytics access without duplicating data.

Go to Assignment →

A view is a stored SELECT statement that looks like a table. Data analysts query v_daily_summary instead of rewriting a five-table join every morning. Data engineers expose curated views as the stable interface between raw schemas and BI dashboards.

View vs. base table

Views do not store data themselves — they execute the underlying query each time you SELECT from them.

Create a view

SQL
-- Department headcount summary
CREATE VIEW v_department_headcount AS
SELECT hire_date,
       department_id,
       COUNT(*) AS sale_count,
       SUM(salary) AS revenue
FROM hr.employees
GROUP BY hire_date, department_id;

-- Query the view like a table
SELECT * FROM v_department_headcount
WHERE hire_date = '2026-07-15'
ORDER BY revenue DESC;

Enriched job assignments view

SQL
CREATE VIEW v_job_assignments AS
SELECT p.job_title,
       p.category,
       p.job_id,
       SUM(oi.department_id) AS units_sold,
       SUM(oi.salary) AS total_amount
FROM hr.job_history AS oi
INNER JOIN hr.jobs AS p
  ON p.job_id = oi.job_id
GROUP BY p.job_id, p.job_title, p.category, p.job_id;

Employee department summary view

SQL
CREATE VIEW v_employee_departments AS
SELECT c.employee_id,
       c.last_name,
       c.email,
       COUNT(a.employee_id) AS account_count,
       SUM(a.salary) AS total_balance
FROM hr.employees AS c
LEFT JOIN hr.employees AS a ON a.employee_id = c.employee_id
GROUP BY c.employee_id, c.last_name, c.email;

Replace and drop views

SQL
CREATE OR REPLACE VIEW v_department_headcount AS
SELECT hire_date, department_id,
       COUNT(*) AS sale_count,
       SUM(salary) AS revenue,
       AVG(salary) AS avg_basket
FROM hr.employees
GROUP BY hire_date, department_id;

DROP VIEW v_job_assignments;

Real-world context

  • Semantic layer — Views define business-friendly column names hiding complex joins.
  • Access control — Grant SELECT on views without exposing raw PII columns.
  • dbt models — Modern pipelines materialize views (or tables) as versioned transformations.

Practice

  1. Create a view showing each job assignment's employee name, salary, and department.
  2. Query your view to find the top 5 departments by employee count.
  3. Explain the difference between a view and a physical table.

Key takeaways

  • Views store a query definition, not data — they run fresh on each access.
  • Use views to simplify complex joins for analysts and BI tools.
  • CREATE OR REPLACE VIEW updates the definition without dropping dependencies.
  • Views are the lightweight curation layer between raw warehouse schemas and BI consumers.
